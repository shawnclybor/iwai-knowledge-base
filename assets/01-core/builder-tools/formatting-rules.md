# KB File Formatting Rules

This document defines the required structure, metadata, and style for all knowledge base files generated by this builder project. Every KB file must follow these rules so the deliverable Claude Project can retrieve information reliably.

---

## File Format

All KB files are markdown (`.md`). No other formats.

---

## Required Metadata Block

Every KB file starts with a YAML-style metadata block at the top, fenced by `---`:

```
---
file: [filename].md
workflow: [workflow name or number]
version: 1.0
last_updated: [YYYY-MM-DD]
primary_sources: [comma-separated list of source files used]
owner: [who is responsible for keeping this file current]
review_cadence: [monthly | quarterly | annually]
---
```

Do not skip any fields. If a value is unknown, use `TBD`.

---

## Heading Hierarchy

Use consistent heading levels across all files:

```
# File Title (H1) — one per file, matches the file purpose
## Major Section (H2) — top-level categories
### Subsection (H3) — specific topics within a category
#### Detail (H4) — use sparingly, only when a subsection needs further breakdown
```

Do not skip heading levels (no jumping from H2 to H4). Do not use H1 for anything other than the file title.

---

## Region-Specific Labels

When content varies by region (state, province, country, etc.), use a consistent labeling pattern.

For inline differences within a paragraph:
```
The filing deadline is [Region A: 30 days] [Region B: 45 days] [Region C: 60 days].
```

For longer region-specific sections, use H3 or H4 headers with region codes:

```
### Filing Requirements

#### NY — New York
[NY-specific content]

#### CA — California
[CA-specific content]
```

Use standard abbreviations for the client's operating regions. Spell out the full name on first use within a section, then abbreviate.

---

## Industry / Category Tags

When content is specific to a particular industry, client segment, or category, tag it clearly:

```
**Industry:** [Tag 1] | [Tag 2] | General
```

Place tags at the beginning of a section or subsection that is category-specific. Content that applies universally uses the `General` tag or no tag.

---

## Inline Source Citations

Every substantive claim, process description, or factual statement must cite where it came from:

```
Consultants spend approximately 20% of billable time on recurring questions. [Source: stakeholder-interview-transcript-2025-01-13.txt]
```

For information synthesized from multiple sources:
```
[Sources: stakeholder-interview-transcript-2025-01-13.txt, storage-inventory.md]
```

For information attributed to a specific person during interviews:
```
[Source: stakeholder-interview-transcript-2025-01-13.txt — Jane Smith]
```

---

## Content Gap Markers

When source materials don't contain enough information to fully populate a section, mark it:

```
[CONTENT GAP — requires follow-up with [person or role]]
```

Or for specific missing data:
```
[CONTENT GAP — exact client count per segment not available in discovery files]
```

Do not fill gaps with fabricated content. Gaps are expected and useful — they tell the consultant what still needs to be gathered from the client.

---

## Placeholder Markers

When realistic example data is needed but specific real data isn't available:

```
[PLACEHOLDER — replace with actual client name]
[PLACEHOLDER — insert current minimum wage as of delivery date]
```

---

## Tables

Use markdown tables for structured reference data. Always include a header row:

```
| Region | Requirement | Deadline | Notes |
|---|---|---|---|
| Region A | Annual filing | March 31 | Applies to firms with 10+ employees |
| Region B | Biannual filing | June 30 / Dec 31 | No size threshold |
```

For large reference tables (20+ rows), consider splitting into separate subsections by category rather than one massive table.

---

## Lists

Use bullet points for unordered information. Use numbered lists only for sequential steps or ranked items.

Keep list items to 1–2 sentences. If a list item needs more explanation, make it a subsection instead.

---

## Cross-References Between KB Files

When one KB file references information that lives in another, use this format:

```
For the step-by-step compliance audit process, see `workflows-and-processes.md` > Compliance Audit Workflow.
```

Always reference the specific section, not just the file name.

---

## Tone and Voice

- Professional but approachable — these files are read by the client's team, not their customers
- Direct and actionable — lead with the answer, then provide context
- No jargon without explanation — define technical terms on first use
- Avoid hedging language ("might," "could possibly," "it depends") — if the answer varies, state the conditions explicitly

---

## File Length Guidelines

- Aim for 500–1500 words per KB file for operational and reference files
- Educational or onboarding files may be longer (up to 2500 words)
- If a file exceeds 2500 words, consider splitting it into sub-files
- Shorter is better if completeness is maintained — Claude retrieves more reliably from focused, well-structured files than from sprawling documents

---

## Do Not Include

- Raw interview quotes longer than one sentence (paraphrase and cite instead)
- Redundant information that exists in another KB file (cross-reference instead)
- Legal opinions or definitive legal advice — frame compliance information as general guidance with a note to verify with legal counsel for edge cases
- Client-identifying information — use industry/type references, not specific client names
