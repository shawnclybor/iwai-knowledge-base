# CLAUDE.md

## Project Overview

This repo contains lesson materials for IWAI's "Build it Together" (BiT) series on Knowledge Bases. The lesson teaches AI consultants how to build a Claude Project-based knowledge base for a client, using a fictional case study: **Truepoint HR Solutions** (Charlotte, NC — 35 employees, ~120 clients, ~$2.8M revenue, HR consulting).

The core lesson slides and case study live in the attached project knowledge base (`Build-it-Together-Knowledge-Bases-Core-Workflow.md`), not in this repo.

## Architecture: Two Claude Projects

The lesson demonstrates a two-project pattern:

- **Project A (Builder):** The consultant's processing project. Builder tools (core instructions) and processing templates are uploaded to this project. Raw discovery materials stay in the client's cloud storage and are accessed via MCP connector (Google Drive, SharePoint, etc.). Claude reads the raw materials through MCP, processes them through the templates in sequence, and generates structured KB files. The builder tools and templates are **project-agnostic** — they work for any client, not just Truepoint.
- **Project B (Deliverable):** The client-facing project installed for the client. Contains generated KB files + its own claude.md + MCP connectors. This is what the client's team uses daily.

The raw client files in this repo (`01-core/raw-client-files/`) are lesson materials that represent what would live in a client's Google Drive. In an actual engagement, these would not be uploaded to the builder project — they'd be accessed via MCP.

The deliverables folder (`01-core/deliverables/`) is populated by running raw files through the builder tools, not created manually.

## Builder Workflow

The builder project processes raw materials through templates in a defined sequence:

```
Step 1:  Locate raw materials (transcripts, sample docs, Drive access)
Step 2a: Process transcripts → structured data
Step 2b: Document storage → folder/file inventory
Step 2c: Audit content → KUAD tags (Keep / Update / Archive / Delete)
Step 2d: Compile recurring questions → categorized question list
Step 2e: Identify knowledge gaps → gap list with impact ratings
Step 3:  Source assessment → proposed KB files (wait for approval)
Step 4:  Generate KB files
Step 5:  Review and handoff
```

Raw client files (`raw-client-files/`) are inputs to Step 1. Everything from Step 2 onward is generated by the builder.

## Key Concepts

- **KUAD tagging:** Keep/Update/Archive/Delete classification applied during the content audit step (2c) to determine which documents to trust during KB generation
- **Builder tools are agnostic:** All files in `builder-tools/` and `templates/` contain no Truepoint-specific references — they're reusable across any client engagement
- **Raw client files are Truepoint-specific:** The transcripts and sample files are written for the Truepoint case study
- **MCP is required for the builder project:** The builder project accesses raw client materials through MCP — nothing is uploaded except the builder tools and templates. MCP is read-only: for pulling file contents during processing, not for organizing or modifying Drive
- **Templates produce intermediate outputs:** The processing templates (Step 2a–2e) generate intermediate files in conversation (inventories, audits, question lists, gap analyses) that feed into KB file generation

## File Conventions

- All KB output files use markdown (`.md`)
- All intermediate outputs (inventories, audits, question lists, gap analyses) are markdown
- The stakeholder interview transcripts use Otter.ai format (`.txt`) with named speakers
- Sample client files include "Document Notes" sections tracing template sources and auditor flags
